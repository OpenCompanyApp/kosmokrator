KosmoKrator settings are schema-backed. The schema drives `/settings`, `settings:*`, validation, type parsing, choices, source reporting, and effect timing.

Use this page when you need an exhaustive list. Use [Configuration](/docs/configuration) for concepts and examples.

## Scopes And Precedence

Settings can be written to two user-editable scopes:

| Scope | Path | Use for |
| --- | --- | --- |
| Global | `~/.kosmo/config.yaml` | Personal defaults, provider/model choice, secrets references, UI preference. |
| Project | `.kosmo/config.yaml` or `.kosmo.yaml` | Repository policy, team-safe permissions, project model overrides. |

Effective precedence is:

```text
project override -> global override -> bundled default
```

The CLI reports both the source and the effective value:

```bash
kosmo settings:get agent.default_provider --json
kosmo settings:list --json
```

## Type Parsing

| Type | Accepted input |
| --- | --- |
| `toggle` | `on`, `off`, `true`, `false`, `1`, `0`, `yes`, `no`. |
| `choice` | One of the option values from `settings:options`. |
| `dynamic_choice` | A value discovered from runtime catalogs, such as provider/model IDs. |
| `number` | Integer or decimal depending on the setting. |
| `text` | Raw string. |
| `string_list` | Comma-separated, whitespace-separated, newline-separated, or JSON array. |

Batch apply JSON when setting many values:

```bash
jq -n '{
  settings: {
    "agent.mode": "plan",
    "tools.default_permission_mode": "guardian",
    "context.auto_compact": "on"
  }
}' | kosmo settings:apply --stdin-json --project --json
```

## Effect Timing

| Effect | Meaning |
| --- | --- |
| `applies_now` | Active runtime can observe it immediately or on the next tool/prompt boundary. |
| `next_turn` | The next user prompt or LLM turn will use it. |
| `next_session` | Restart the session to guarantee the new value is wired into all services. |

| --- | --- | --- | --- | --- |

```bash
kosmo settings:options agent.default_provider --json
```

## Integrations

| Setting | Type | Default | Effect | Description |
| --- | --- | --- | --- | --- |
| `integrations.permissions_default` | choice: `allow`, `ask`, `deny` | `ask` | applies now | Default read/write policy for integration calls without provider-specific policy. |

Integration activation, credentials, named accounts, and provider read/write policies are configured with `integrations:configure`, not through generic `settings:set`, because the command knows each provider credential schema.

```bash
kosmo integrations:configure plane --account production \
  --set api_key="$PLANE_API_KEY" \
  --set workspace_slug=my-workspace \
  --enable \
  --read allow \
  --write ask \
  --json
```

## Discovery Commands For Agents

Agents and wrappers should discover rather than hard-code:

```bash
kosmo settings:list --json
kosmo settings:list --category web --json
kosmo settings:get web.search.enabled --json
kosmo settings:options tools.default_permission_mode --json
kosmo settings:options agent.default_model --provider anthropic --json
kosmo settings:doctor --json
```

When adding new settings in code, update this page, [Configuration](/docs/configuration), and the website navigation if a new category deserves its own guide.
